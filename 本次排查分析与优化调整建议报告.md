# 本次排查分析与优化调整建议报告

## 1. 背景与问题概述

本次排查围绕以下几类现象展开：

1. sandbox 写文件失败，日志中出现容器启动异常、网络不可达等报错。
2. 前端历史会话列表消失，刷新页面后出现 hydration mismatch 报错。
3. 新聊天页向 agent 发送消息时，请求 `http://localhost:2026/api/langgraph/threads` 长时间无响应，页面持续转圈。
4. 用户关注当前系统到底使用 `.deer-flow` / sqlite 还是 postgres 持久化，以及为什么已有线程数据却无法在前端展示。

本报告基于本次实际代码阅读、日志分析、容器检查、数据库查询和接口验证结果整理，目标是给出：

- 现象与根因的对应关系
- 当前架构中的关键断点
- 可落地的短期止血方案
- 中长期优化与调整建议

## 2. 本次排查范围

本次排查未修改任何代码，也未创建除本报告外的其他文件。主要工作包括：

- 检查启动链路：`Makefile`、`scripts/docker.sh`、`docker-compose-dev.yaml`
- 检查配置：`config.yaml`
- 检查 sandbox 实现：AIO sandbox provider、local backend
- 检查前端线程列表与聊天发送逻辑
- 检查 gateway 线程搜索逻辑
- 检查 nginx 路由转发配置
- 检查 langgraph / gateway 日志
- 检查 postgres 中 checkpoints / store 数据
- 在容器内直接验证 LangGraph `/threads` 与 checkpointer 行为

## 3. 关键结论摘要

### 3.1 历史会话列表为空的直接原因

前端历史会话列表当前依赖的是 LangGraph API：

- 前端默认请求 `/api/langgraph/threads/search`
- 不是 gateway 的 `/api/threads/search`

而当前 LangGraph 侧的线程列表能力存在异常，导致：

- `/api/langgraph/threads/search` 返回空或超时
- 前端 `RecentChatList` 在 `threads.length === 0` 时直接不渲染
- 因此用户看到“历史会话列表消失”

### 3.2 新聊天页一直转圈的直接原因

新聊天页发消息时，前端会先依赖 LangGraph 的线程创建能力：

- 请求 `POST /api/langgraph/threads`
- 该请求当前会卡住
- 前端因此一直处于 loading 状态

已经确认：

- 不是前端单独问题
- 不是 nginx 转发问题
- 是 LangGraph 服务自身 `/threads` 路由卡住

### 3.3 LangGraph `/threads` 卡住的核心根因

当前 LangGraph 使用的是自定义 Postgres checkpointer：

- 日志显示：`Using custom checkpointer: AsyncPostgresSaver`

进一步验证发现：

- `aget_tuple(thread_id)` 可快速返回，说明 postgres 可用、按 thread_id 精确读取正常
- 但 `alist(None)` 明显异常，表现为超时甚至被系统杀掉

这说明：

- LangGraph 在线程列表相关能力中依赖 `alist(None)`
- 当前自定义 checkpointer 在这条路径上不可用或性能极差
- 这正是 `/threads` 与 `/threads/search` 卡住的核心原因

### 3.4 sandbox 不是当前“转圈”的主因

sandbox 这条链路确实出现过：

- 容器名冲突
- `Server disconnected without sending a response`
- `Network is unreachable`

但结合最新日志可确认：

- 本次相关 run 中 sandbox 已成功创建
- LangGraph 已能访问 `http://host.docker.internal:8081`
- `/v1/file/read`、`/v1/shell/exec` 已返回 `200 OK`

因此：

- sandbox 仍有稳定性问题
- 但不是当前这次“新聊天页一直转圈”的主因
- 当前主因仍然是 LangGraph `/threads` 卡住

## 4. 详细分析

### 4.1 `make docker-start` 是否会启动 sandbox 容器

结论：**不会以 compose 常驻服务方式启动 sandbox 容器。**

当前配置下：

- `sandbox.use = deerflow.community.aio_sandbox:AioSandboxProvider`
- `sandbox.image = np-all-in-one-sandbox:latest`

AIO 模式下：

- `make docker-start` 只会启动主服务，如 frontend / gateway / langgraph / nginx
- sandbox 容器由 langgraph 或 gateway 进程通过 docker socket 按需动态创建
- 因此“docker 里看不到 sandbox 容器”本身不等于异常

### 4.2 当前系统主要使用哪种持久化

结论：**当前运行时主要使用 postgres，不是 sqlite。**

依据：

- `config.yaml` 中 `checkpointer.type=postgres`
- LangGraph 日志显示 `AsyncPostgresSaver`
- Gateway 日志显示 `AsyncPostgresStore`

虽然本地和容器内仍存在：

- `backend/.deer-flow/threads/...`
- `checkpoints.db`

但这些文件并不代表当前主运行链路仍以 sqlite 为主。

### 4.3 为什么“明明有线程数据，前端却看不到”

当前 postgres 中：

- `checkpoints` 表有大量数据
- `store` 表为空
- `checkpoints` 中存在多个 thread_id

说明：

- 线程状态数据并没有丢
- 但线程列表索引没有建立起来

同时，gateway 的 `/api/threads/search` 已实现：

- store 快速路径
- checkpointer 补扫逻辑

但前端当前没有走 gateway，而是走 LangGraph 原生 `/api/langgraph/threads/search`。

因此出现了一个关键断层：

- 数据在 checkpointer 里
- gateway 有补扫能力
- 但前端没走 gateway
- LangGraph 自己的线程列表又卡在 `alist(None)`
- 最终前端看起来像“没有历史会话”

### 4.4 为什么 LangGraph `/threads` 会卡住

本次验证结果表明：

1. `aget_tuple` 正常
   - 说明数据库连接、基础读能力正常
2. `alist(None)` 异常
   - 3 秒内拿不到第一条
   - 放开执行后甚至被系统杀掉

这说明问题不在：

- postgres 是否可连接
- thread_id 是否存在
- nginx 是否转发正确

而在：

- LangGraph 线程列表相关能力依赖 `alist(None)`
- 当前自定义 checkpointer 在该路径上存在严重性能或兼容性问题

### 4.5 sandbox 容器名冲突的判断

当前 sandbox 使用确定性 `sandbox_id`：

- 同一 thread_id 会映射到同一个 sandbox_id
- 容器名形如 `deer-flow-sandbox-<sandbox_id>`

日志显示：

- 同一时刻有多个线程尝试为同一 thread 创建 sandbox
- 其中一个 `docker run` 成功
- 另一个因容器名已存在而报冲突
- 随后代码通过 discover 机制发现已有容器并复用

因此：

- 容器名冲突是并发创建过程中的竞争现象
- 当前实现已经具备一定自恢复能力
- 它不是当前主故障，但说明 sandbox 并发创建路径仍有优化空间

### 4.6 `Network is unreachable` 的判断

历史日志中确实出现过：

- `Failed to execute command in sandbox: [Errno 101] Network is unreachable`

但本次最新 run 中又能看到：

- `host.docker.internal` 可解析
- sandbox 接口请求返回 `200 OK`

因此更合理的判断是：

- 该问题具有间歇性
- 可能与容器启动窗口期、网络栈状态、IPv4/IPv6 选择有关
- 目前不是阻塞主链路的第一优先级问题

## 5. 问题优先级排序

建议按以下优先级处理：

### P0：恢复线程列表与新聊天可用性

目标：

- 历史会话列表恢复展示
- 新聊天页不再一直转圈

当前最直接的阻塞点是：

- LangGraph `/threads`
- LangGraph `/threads/search`

### P1：修复 LangGraph 线程列表能力

目标：

- 让 LangGraph 原生线程接口恢复可用
- 避免前端长期依赖兼容绕行方案

### P2：提升 sandbox 稳定性

目标：

- 降低并发创建冲突
- 降低偶发网络不可达
- 提升工具调用稳定性

### P3：处理 hydration mismatch

目标：

- 消除前端刷新时的 SSR/CSR 不一致报错

该问题影响体验，但不是当前主链路阻塞项。

## 6. 优化与调整建议

## 6.1 短期止血建议

### 建议一：前端线程相关请求优先切到 gateway 兼容层

建议将前端默认 LangGraph base URL 从：

- `/api/langgraph`

调整为可兼容 gateway 的路径，例如：

- `/api/langgraph-compat`

这样做的收益：

- 历史线程列表可复用 gateway 已有的 store + checkpointer 补扫逻辑
- 可以绕开 LangGraph 原生 `/threads` 当前的卡死问题
- 能更快恢复用户可见功能

适用场景：

- 需要先恢复可用性
- 接受短期通过兼容层绕行

### 建议二：优先验证 gateway 兼容接口是否覆盖当前前端所需能力

在切换前，需要确认：

- `threads.search`
- `threads.create`
- `threads.update`
- `runs.stream` / `runs.joinStream`

这些前端当前依赖的能力，gateway 兼容层是否都已覆盖。

如果覆盖完整，可优先切换；如果不完整，则需要补齐兼容实现。

## 6.2 中期修复建议

### 建议三：专项排查 `AsyncPostgresSaver.alist(None)`

这是当前最关键的根治方向。

建议重点检查：

1. LangGraph 当前版本对自定义 checkpointer 的 `alist` 调用方式
2. `AsyncPostgresSaver.alist` 在当前数据量下的 SQL 行为
3. 是否存在缺少索引、全表扫描、排序代价过高等问题
4. 是否存在 namespace / checkpoint 过滤条件不合理的问题
5. 是否存在当前 adapter 与 LangGraph 版本之间的兼容性问题

建议输出专项结论：

- 是 SQL 性能问题
- 是 LangGraph adapter 兼容问题
- 还是当前数据结构导致的遍历异常

### 建议四：为线程列表建立显式索引层，不再依赖全量 checkpoint 扫描

从架构角度看，线程列表不应长期依赖 checkpointer 全量扫描。

建议：

- 将 thread 元数据显式写入 store 表
- 或建立独立 thread index 表
- 在线程创建、运行、更新标题、更新时间时同步维护索引

这样可以避免：

- 每次列表查询都扫 checkpoint
- checkpointer 成为列表接口瓶颈
- LangGraph 与 gateway 在“线程列表来源”上出现分裂

### 建议五：统一线程列表的权威来源

当前存在两个来源：

- LangGraph 原生线程接口
- Gateway 自己的线程聚合逻辑

建议明确一个权威来源：

- 要么前端统一走 gateway
- 要么 LangGraph 侧补齐稳定线程索引能力

不建议长期维持“双来源并存但行为不一致”的状态。

## 6.3 sandbox 优化建议

### 建议六：继续收敛同一 thread 的 sandbox 并发创建窗口

虽然当前已有：

- 确定性 sandbox_id
- 文件锁
- discover 机制
- warm pool

但日志仍显示同一 thread 存在近乎同时的 `docker run` 尝试。

建议继续检查：

- acquire 路径是否存在多实例/多线程重复进入窗口
- 文件锁粒度是否足够早
- discover 前后是否仍存在竞态窗口

### 建议七：对 `host.docker.internal` 链路增加更强健的探测与回退

建议补充：

- sandbox 创建后更明确的健康检查
- IPv4 优先或可配置策略
- 对短暂网络不可达增加有限重试
- 对失败原因做更细粒度日志区分

这样可以降低偶发网络问题对工具调用的影响。

## 6.4 前端体验建议

### 建议八：为线程列表与线程创建增加超时和降级提示

当前用户看到的是：

- 历史列表直接消失
- 新聊天页一直转圈

建议前端增加：

- 线程列表请求超时提示
- 线程创建失败提示
- 区分“无数据”和“接口异常”两种状态
- 对长时间 pending 的请求给出可见错误反馈

这样即使后端异常，用户也不会误以为“数据全没了”。

### 建议九：单独处理 hydration mismatch

当前 hydration mismatch 更像是 `CommandPalette` / Radix Dialog 的 SSR/CSR 不一致问题。

建议后续单独处理：

- 检查是否需要延迟挂载相关组件
- 检查是否需要避免首屏 SSR 输出与客户端状态不一致

该问题不影响本次主链路根因判断，但建议后续修复以减少噪音。

## 7. 推荐执行顺序

建议按以下顺序推进：

1. **先恢复可用性**
   - 评估并切换前端线程相关请求到 gateway 兼容层
2. **再做根因修复**
   - 专项排查 `alist(None)` 卡死问题
3. **补齐线程索引机制**
   - 建立稳定的 thread index，而不是依赖 checkpoint 全量扫描
4. **最后处理稳定性与体验问题**
   - sandbox 并发创建优化
   - 网络不可达优化
   - hydration mismatch 修复
   - 前端错误提示完善

## 8. 最终结论

本次排查最核心的结论是：

1. 当前主故障不在前端，也不在 nginx。
2. 当前主故障是 **LangGraph 原生 `/threads` / `/threads/search` 能力异常**。
3. 更底层的关键根因是：**自定义 Postgres checkpointer 的 `alist(None)` 路径不可用或性能极差**。
4. 历史会话列表消失与新聊天页一直转圈，本质上都由这一问题链引发。
5. sandbox 确实存在并发创建冲突和历史网络异常，但从本次最新日志看，它不是当前主阻塞项。

因此，建议策略应分为两步：

- **短期先绕开 LangGraph 原生线程接口，恢复用户可用性**
- **中期再根治 `alist(None)` 与线程索引机制问题**

---

如需继续推进，下一步建议优先选择以下两条路径之一：

1. 直接实施“前端改走 gateway 兼容层”的调整方案
2. 继续深挖 LangGraph `/threads` 内部调用链，定位 `alist(None)` 的具体阻塞点
