# np-ppt-generation

这个 skill 用来在 JavaScript 中通过 `pptxgenjs` 构建或修改**可编辑**的 PowerPoint 文档。

它面向通用 agent 环境设计，不假设存在 Codex 专用运行时，也不假设某些依赖已经预装。环境里如果缺依赖，agent 应主动补齐，而不是直接放弃。

## 什么时候该用

当用户希望你做这些事情时，使用本 skill：

- 新建 `.pptx`
- 修改现有 PPT
- 把 HTML、Markdown、会议纪要、策划案、表格分析转成 PPT
- 根据截图、参考 PDF、旧 deck 复刻一版可编辑 PPT
- 检查幻灯片是否溢出、排版是否碰撞、字体是否替换

## 依赖策略

默认认为环境“基本可用”，但不保证所有依赖都已经安装。如果缺了，agent 要自己想办法补齐。

### Node 依赖

推荐依赖：

- `pptxgenjs`
- `skia-canvas`
- `linebreak`
- `fontkit`
- `prismjs`
- `mathjax-full`

注意：`mathjax-full` 不要假设可以直接 `require("mathjax-full")`。应优先使用子路径导入。当前 helper 也是这样写的：

- `mathjax-full/js/mathjax.js`
- `mathjax-full/js/input/tex.js`
- `mathjax-full/js/output/svg.js`

### Python 依赖

用于渲染和校验：

- `pdf2image`
- `Pillow`
- `python-pptx`
- `numpy`

### 系统命令

用于 PPTX -> PDF -> PNG 渲染链，以及字体检查：

- `soffice` / LibreOffice
- `pdfinfo`
- `pdftoppm`
- `fc-list`

可选但常有帮助：

- `inkscape`
- `imagemagick`
- `ghostscript`

## 安装建议

如果当前工作目录里缺 Node 依赖，可以直接在工作目录安装：

```bash
npm install pptxgenjs skia-canvas linebreak fontkit prismjs mathjax-full
```

如果 Python 侧缺依赖：

```bash
python3 -m pip install pdf2image Pillow python-pptx numpy
```

如果渲染脚本报系统命令缺失，要补这些：

- LibreOffice / `soffice`
- Poppler（至少要 `pdfinfo` 和 `pdftoppm`）

## 标准工作流

1. 先判断任务是“新建 deck”“修改 deck”还是“复刻参考稿”。
2. 先做简短规划，再开始画页：
   - 明确受众、汇报目标、叙事顺序、页数结构。
   - 明确视觉系统：背景基调、主色/辅助色、标题风格、卡片风格、图表风格、装饰语言。
   - 明确哪些内容属于可编辑文本层，哪些内容属于视觉层。
3. 在当前任务下创建独立工作目录。
4. 把 `assets/pptxgenjs_helpers/` 复制到工作目录，并优先从本地副本导入。
5. 使用 `pptxgenjs` 在 JavaScript 中构建 PPT。
6. 先把版式和主题做完整，再保留 PowerPoint 原生可编辑对象：文字、表格、形状、简单图表。
7. 如需图片或主视觉，先决定每页视觉资产策略：整页背景、侧边图、局部 hero、纹理条、角标、装饰块，不能“有图就塞进去”。
8. 生成后必须渲染成 PNG 做视觉复核，不能只看源码或只看命令成功与否。
9. 必须逐页打开并查看单页渲染图，按页号从头到尾检查，不能跳页、不能只抽查，不能只看 montage。
10. 页面密度高或边界紧时，必须跑 overflow 和字体检查。
11. 修完问题后必须重新渲染并再次逐页复核，直到达到可交付标准。
12. 最终交付 `.pptx`、源 `.js`，以及必要的重建素材。

## 写 PPT 的规则

- 先明确页面尺寸。默认使用 `LAYOUT_WIDE`，除非源材料明显不是 16:9。
- 显式设置主题字体，不依赖 PowerPoint 默认字体。
- 优先使用可编辑的文字、表格、形状和原生图表，而不是整页截图。
- 所有用户会修改的标题、正文、数字、标签、表格内容、图表标签，都必须保留为可编辑对象，不要烘进图片。
- 图片放置优先用 helper，而不是随手硬算裁切。
- 列表请用 bullet 配置，不要直接手打 `•`。
- 只有真的需要公式时才用 `latexToSvgDataUri()`。
- 对于新生成或大改的 deck，源码里应保留：
  - `warnIfSlideHasOverlaps(slide, pptx)`
  - `warnIfSlideElementsOutOfBounds(slide, pptx)`
- 如果某些 overlap 告警是“容器 shape + 内部文字”这种预期行为，可以用 debug 开关包起来，或在对应代码处写一个短注释说明。
- 页面上不要出现实现过程语言、元话术或面向制作者的说明，例如：
  - “本页”
  - “本 deck”
  - “AI 生成”
  - “自动生成”
  - “可编辑 PPT”
  - “已验证输出”
  除非用户明确要求做方法论说明页。

## 文本层与视觉层分离

默认采用“文本层 + 视觉层”双层结构：

- 文本层：
  - 标题
  - 副标题
  - 正文
  - 标签
  - 日期
  - KPI 数字
  - 图表标题和图例
  - 表格文字
  - 结论要点
- 视觉层：
  - 背景色块
  - 装饰几何形
  - 纹理
  - 无文字主视觉图
  - 无文字氛围图
  - 非数据型装饰线条和结构元素

执行时遵守这些规则：

- 不要把真实标题、真实数字、真实表格文字做进图片。
- 如果使用图片，它应该服务于氛围、构图和主题，而不是替代真实文字层。
- 如果图片里自带可读文字、标签、图表数据、Logo、水印，优先换图、裁切或弱化处理，不要把它当成最终可交付视觉层。
- 如果某页信息很多，优先用卡片、面板、分栏、分页来承载文本层，而不是把文字压到复杂背景图上。

## 视觉风格要求

不要把 PPT 做成默认模板味很重的普通商务页。默认应向“已设计过的业务汇报稿”靠拢，风格可参考此前已经做过的那几版会员运营汇报：

- 背景避免纯白大平面，优先使用柔和暖色或低饱和浅色底，如米色、浅砂色、雾蓝灰。
- 使用 1 个主强调色 + 1 到 2 个辅助色，避免整 deck 色彩失控。常见安全组合：
  - 砖红 / 橙金 / 青蓝
  - 深蓝灰 / 米白 / 铜橙
  - 墨蓝 / 浅卡其 / 数据蓝绿
- 标题区、数字卡片区、图表区要有明确层级，不要所有模块长得一样。
- KPI 卡片应有顶部强调条、轻背景面板、统一圆角和清晰数值层级。
- 图表不能只满足“能看”，还要在配色、字号、标签密度上和整页风格一致。
- 尽量让每页都有完整的版式意图：封面页、框架页、数据页、结论页的视觉结构应明显不同。
- 留白要主动设计，不要把内容机械铺满画布。
- 如果没有用户指定品牌规范，优先做成“咨询汇报 / 管理层汇报”风格，而不是教学课件风格。
- 如果页面已经信息密集，优先删减、重组、拆页，不要一味缩小字号。

### 视觉系统规划

开始做页之前，先在脑子里或工作记录里回答这几件事：

- 这套 deck 的受众是谁，偏管理层、业务团队还是执行团队。
- 语气更像“战略判断”“经营分析”还是“项目复盘”。
- 视觉基调是稳重、温和、科技感、零售感还是增长感。
- 标题系统是否偏强对比，还是偏理性克制。
- 卡片系统是浅底描边，还是深浅对比分区。
- 图表系统是否强调数据色，还是强调中性背景下的重点高亮。
- 这套 deck 的装饰语言是什么：色块、几何切角、细分隔线、圆角卡片、局部纹理、轻阴影。

如果这些没想清楚就开始堆内容，最终很容易变成“内容是对的，但看起来像没设计过”。

### 字体系统建议

如果没有品牌指定字体，默认要有清晰的字体分工：

- 标题字体要更有存在感。
- 正文字体要稳、易读、适合长段说明。
- 数字和 KPI 可以比正文更醒目，但不能和标题抢层级。

建议默认策略：

- 中文环境：
  - 标题优先用较有力度的无衬线字体
  - 正文优先用稳健清晰的无衬线字体
- 英文或混排环境：
  - 可参考 `Poppins` 做标题
  - `Lato` 做正文
  - 只有在正式、品牌感很强的稿件中才考虑 serif

无论用哪套字体，都要确保：

- 标题、正文、数字、图表标签之间有明确层级。
- 整 deck 不要随意混入太多字体。
- 图表、表格、页脚也要服从同一套字体系统。

默认可采用的视觉基线：

- 标题字体感觉要更强，正文要稳，数字要最醒目。
- 卡片和面板使用轻描边或低对比边框，不要过重黑线。
- 图表背景尽量干净，网格线弱化，标签字体统一。
- 中文页建议避免过度花哨装饰，用少量背景几何形、色块和层次阴影即可。

### 推荐主题配方

如果用户没有指定品牌视觉，优先从下面这类“会员运营汇报风格”里选一套，并在整 deck 保持一致，不要每页换一套审美：

- 暖米白底 + 墨蓝标题 + 铜橙强调：
  - 背景可用 `#F6F1E8`、`#EFE7DA`
  - 主标题/主分隔可用 `#213547`
  - 强调数字、强调条、重点标签可用 `#C46A2D`
- 雾蓝灰底 + 深蓝灰标题 + 青蓝数据色：
  - 背景可用 `#EEF3F4`
  - 主标题可用 `#243746`
  - 数据强调可用 `#2F7E8D`
- 浅卡其底 + 墨蓝标题 + 橙金重点：
  - 背景可用 `#F4EBDD`
  - 主标题可用 `#1F3142`
  - 重点色可用 `#D2872C`

### 页面美化动作

仅仅“排整齐”还不够。默认应主动做这些美化动作，除非用户明确要求极简或已有品牌规范：

- 封面页：
  - 用大标题 + 副标题 + 日期/客户信息的三段层级，不要只是一行标题居中。
  - 给封面增加低对比背景色块、角标、装饰形或局部渐变，让首页一眼能看出主题。
- 目录/框架页：
  - 用编号、阶段卡、进程条或左右分栏做层级，不要只是项目符号列表。
- 数据页：
  - 数字、结论、图表至少分成 2 个层级区域，避免所有内容堆在同一视觉平面。
  - KPI 卡片优先做成“顶部强调条 + 主数字 + 辅助说明”的结构。
- 表格页：
  - 必须压弱表格线，拉开标题行与内容行层级，重点列/重点行要有温和高亮。
  - 表格过密时，优先拆分或改成卡片摘要 + 附表，不要一味缩小字号。
- 结论页：
  - 必须有 1 个主结论区和 2-3 个支撑点，不要只有一大段文字。
  - 结论页要比中间数据页更有“收束感”和“决策感”。
- 全 deck：
  - 用少量统一的圆角、阴影、描边语言，不要这页是硬边，那页是圆角。
  - 同类模块在不同页面应尽量保持同一种视觉组件写法。
  - 如果页面看起来“太像导出的 Excel”或“太像默认模板”，说明还需要继续美化。

### 每页视觉资产策略

每页都要先判断，它更适合哪一种视觉资产结构：

- 纯文本结构页：
  - 主要靠标题、分栏、卡片、图标、色块建立视觉层次。
- 数据主导页：
  - 重点是图表 panel、KPI 卡片、结论摘要，不需要无意义的大图抢注意力。
- 封面/过渡页：
  - 可以有更强的背景图、氛围图、hero image 或大面积纹理。
- 结论页：
  - 更适合低干扰背景 + 强结论卡片，不适合花哨主图。

如果使用图片，还要提前决定：

- 它是整页背景，还是局部框图。
- 它应该放左边还是右边。
- 哪一侧要留 calm zone 给标题和正文。
- 图片是否会和卡片、表格、标签抢对比度。

不要“最后再想办法塞一张图”，而要在布局阶段就决定图片在页面上的职责。

### 图表与表格设计细则

图表和表格不能只是“放上去能看”，而要融入整套视觉系统。

- 图表：
  - 不接受默认 Office 图表味。
  - 标题、坐标轴、图例、数据标签字体要和整页字体系统一致。
  - 网格线、轴线、分隔线要弱化，重点数据色要更明确。
  - 图表最好放在 panel/card 体系里，而不是裸露地漂在画布上。
  - 如果图表已经很复杂，周围的装饰应减弱，避免双重噪音。
- 表格：
  - 标题行必须有清晰层级，不要像原始 Excel 截图。
  - 数据表尽量减少粗重边框，改用浅分隔、重点色高亮、留白分组。
  - 对管理层阅读不重要的细字段，应优先删减、合并或移到附页。

## 渲染和校验

渲染和校验不是“可选建议”，而是默认交付门槛。只要环境支持，就必须执行。

优先跑这些命令：

```bash
python3 scripts/render_slides.py deck.pptx --output_dir rendered
python3 scripts/slides_test.py deck.pptx
python3 scripts/create_montage.py --input_dir rendered --output_file montage.png
python3 scripts/detect_font.py deck.pptx --json
```

## 逐页复核要求

完成 PPT 生成后，必须做**逐页视觉复核**，不能只抽看 1-2 页。

- 必须逐页打开单页渲染图，至少从第 1 页检查到最后 1 页。
- 必须按页号顺序检查，不能只看 montage 就当完成。montage 只能当总览辅助，不能替代逐页复核。
- 必须在复核过程中明确知道自己当前检查到第几页；页数超过 6 页时，必须写出逐页复核记录。
- “逐页复核”指真的看过每一张单页图，而不是只运行渲染脚本、只看命令成功、只看 montage、只看首尾两页。
- 必须检查标题、正文、图表、表格、页脚、注释框、图片、背景装饰是否有碰撞、裁切、溢出、留白失衡或字体异常。
- 除了功能性错误，还必须检查“视觉完成度”：
  - 是否仍像默认模板
  - 颜色是否单薄或不协调
  - 卡片、图表、表格是否缺少层级
  - 页面是否太空或太满
  - 封面和结论页是否缺乏重点
  - 图表是否仍带明显默认 Office 风格
  - 页面是否像“导出的 Excel”而不是“设计过的汇报稿”
  - 文本层和视觉层是否互相打架
  - 图片是否在错误的位置抢了正文注意力
- 必须把复核结果落成痕迹，且同时满足以下要求：
  - 保留 `rendered/` 目录下的逐页 PNG
  - 生成 montage 总览图
  - 在工作记录或最终说明中明确写出“已逐页复核”
- 必须留一份简短复核记录，至少写出：
  - 总页数
  - 起止页号
  - 是否逐页看完
  - 至少 3 个具体观察点或修改点
  - 如果有返工，说明返工后又从第几页复查到第几页
- 如果发现问题，必须修改源码后重新导出，再重新渲染复核。不能“知道有问题但先交付”。
- 如果环境支持 `slides_test.py` 或字体检查脚本，不能跳过。

## 修后复检要求

每次修版后都要重新检查，而不是默认“改完应该就好了”。

- 任何会影响布局、字号、图表、表格、分页、图片摆放的修改，都必须重新导出。
- 重新导出后，必须重新渲染。
- 重新渲染后，必须重新逐页复核。
- 如果只改了第 1 页，也不能完全跳过其他页；至少要从改动页开始继续顺序检查到最后一页，并确认没有引入全局样式副作用。
- 修后复检时，必须更新复核记录，不能沿用修改前的旧记录冒充本轮检查结果。
- 只有最后一次复核通过，才算完成。

## 交付门槛

以下条件没满足，就不应把 deck 当成最终交付：

- 已成功导出最终 `.pptx`
- 已生成并查看逐页渲染图
- 已完成逐页视觉复核
- 已形成逐页复核记录，而不是只口头声称“检查过”
- 已修掉发现的明显版式问题
- 修版后已经重新渲染并复核
- 页面视觉风格已经达到“像正式汇报稿”，而不是“能打开的半成品模板稿”
- 文字、数字、标签、表格、图表等核心信息仍保持可编辑，而不是被图片替代
- 页面不存在明显默认 Office 图表味、默认模板味或导出 Excel 味
- 如果环境支持 overflow / 字体检查，则这些检查已执行且没有阻塞性交付问题

## 交付要求

正常交付物包括：

- 最终 `.pptx`
- 生成它的源 `.js`

最终回复尽量简短，但不能省略“是否已完成渲染复核”这一事实。如果因为环境限制无法完成完整复核，必须明确说明缺了哪一步。

最终回复最好至少包含这 3 个状态事实：

- `已逐页复核：是 / 否`
- `已修后复检：是 / 否`
- `已完成渲染校验：是 / 否`

如果做过逐页复核，最终回复里还应简要写出：

- `复核页数：第 X 页 - 第 Y 页`
- `复核记录：已保留 / 未保留`
- `本轮主要修正：...`
